# xrun v0.1-bis — Real vast adapter, poller, xrun_hook (closes v0.1 acceptance)

## Overview

Замкнуть v0.1: заменить stub vast-адаптера на реальные subprocess-вызовы `vastai`,
поднять поллер событий/метрик с offset-based tail, и опубликовать Python-пакет
`xrun_hook` (локально editable + готовый wheel). После этого плана `xrun launch
exp/arborust_v7_C.yaml --detach` действительно создаёт инстанс на vast, гонит
тренировку, поллер пишет события и метрики в SQLite, `xrun pull --ckpt best`
синхронизирует чекпоинт. Это удовлетворяет полный список acceptance-критериев
v0.1 из `docs/ROADMAP.md`.

## Context

- **Files involved**: достраиваем `xrun-vast`, добавляем `xrun-poller`, отдельный
  Python-проект `python/xrun_hook/`.
- **Reference docs**: `docs/ARCHITECTURE.md`, `docs/EVENTS.md` (формат stdin для
  hook), `docs/STATE.md` (poll_offsets таблица), `docs/CLI.md`.
- **Внешние зависимости (Rust)**:
  `tokio` (rt-multi-thread, process, sync, time, fs), `which`, `bytes`,
  `humantime`. Dev: `wiremock` для mock vastai, `tokio-test`.
- **Внешние зависимости (Python hook)**:
  `python_requires = >=3.9`, без runtime-deps (только stdlib + `orjson` опционально).
  Build via `hatchling`. Метаданные: name=`xrun_hook`, version=0.1.0, license=MIT.
- **Process-модель поллера** (фиксируем здесь, в docs нет):
  - При `xrun launch` без `--detach` — поллер живёт в этом же процессе, выводит
    стадии в stderr, exit при `done`/`failed`.
  - При `--detach` — `xrun launch` форкает себя через `xrun __poll-daemon <run-id>`
    (внутренняя hidden-команда), родитель печатает run-id и выходит.
  - При `xrun ls`/`xrun show`/`xrun events` — если для активного run нет живого
    поллера (lock-файл `~/.local/share/xrun/runs/<id>/poller.pid` без процесса),
    стартует poller-задачу в фоне на 30 сек на пере-poll, потом затихает.
  - Lock-файл предотвращает двойной поллинг одного run.
- **Конкуррентность поллера**: глобальный `Semaphore::new(4)` на все subprocess
  вызовы `vastai` — защита от рейт-лимита.
- **vastai retry**: экспоненциальный бэкофф 1s/2s/4s/8s, max 4 попытки на
  идемпотентные операции (`search`, `show`, `execute "tail -c+N"`, `copy`).
  Не-идемпотентные (`create instance`, `destroy`) — 1 попытка + явная ошибка.
- **vastai SSH key requirement**: документируется в `xrun doctor` — если
  `vastai show user` не показывает ssh keys, doctor печатает actionable hint.
- **xrun_hook доставка**: editable install для разработки + wheel в
  `python/xrun_hook/dist/`. Адаптер vast добавляет `pip install xrun_hook` в
  `setup` фазе автоматически (опция `policy.skip_hook_install: bool = false`
  в манифесте — для случаев когда хук уже в кастомном image).

  - `class stage` (контекстный менеджер): на enter пишет `start`, на exit без exc — `ok`, на exit с exc — `fail` + `extra={"error": repr(exc)}`;

  - `done()` — пишет `stage="done" status="ok"`, флашит, закрывает.

## Anti-patterns

- Не реализовывать `policy.on_stage_failed: reprovision` в v0.1 — слишком много сценариев.
- Не использовать nested `tokio::spawn` без явного `JoinSet` — leaks.
- Не парсить stdout vastai как plain text там, где есть `--raw`/`--json` флаги — всегда JSON.
- Не вызывать `vastai destroy` без проверки status — игнорируй `NotFound` ошибки (instance уже мёртв).
- Не писать в `events.jsonl` из rank>0 — JSON line interleaving поломает поллер.
