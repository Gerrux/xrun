# xrun v0.2.3 — Budget guards: idle/lifetime caps, billable confirms, spend dashboard

## Overview

Защитить пользователя от перерасхода на Vast.ai по трём осям:

1. **Забытые инстансы** — hard caps на lifetime/cost/idle, поллер сам сносит
   инстанс при превышении. Цель: при любой ошибке (упавший скрипт, потерянный
   ssh, забытый `--detach`) потери ограничены конкретным числом долларов.
2. **Подтверждение перед billable действиями** — единый confirm-flow перед
   `vast create` / `xrun launch`, видна оценка стоимости при cap-времени и
   текущий баланс. CLI (`--yes` для скриптов), TUI (modal).
3. **Видимость трат** — per-instance аккумулятор `accumulated_cost`, агрегаты
   day/week/month, dashboard-карточка `Today: $X · Burn: $Y/hr · Balance: $Z`,
   soft-alert при дневном бюджете.

## Development Approach

- **Pure budget logic** в `xrun-core/src/budget.rs` без I/O, тестируется
  таблицей кейсов (idle/lifetime/cost cap по отдельности и в сочетании).
- **Hard cap = auto-destroy**, soft cap = только событие в `events.jsonl` +
  status в TUI. Дневной/месячный бюджеты — soft по умолчанию, hard опционально.
- **Идемпотентность destroy**: `auto_destroyed_reason` пишется до `destroy()`,
  при рестарте daemon видит уже-убитые и не дёргает API повторно.
- **Confirm в CLI**: одна функция `confirm_billable_or_exit(estimate)` —
  печатает таблицу (gpu, $/hr, est total at cap, balance), читает stdin
  если tty, fail-loud если non-tty без `--yes`.
- **Confirm в TUI**: новый `Modal::ConfirmBillable { estimate, on_confirm }`,
  Enter/y подтверждает, Esc отменяет; для high-risk (>$2/hr) требует ввод
  части `instance_id` или offer_id.

## Stages

### Stage 1 — Schema + budget core (xrun-core)

- Миграция `003_budget.sql`: `ALTER TABLE instances ADD COLUMN
  max_lifetime_secs INTEGER, max_cost_usd REAL, idle_timeout_secs INTEGER,
  accumulated_cost REAL DEFAULT 0, last_active_at INTEGER,
  auto_destroyed_reason TEXT`. Bump `schema_version=3`.
- `BudgetConfig` в `global.rs`:
  ```rust
  pub struct BudgetConfig {
      pub max_lifetime_hours: f64,         // default 8
      pub max_cost_per_instance_usd: f64,  // default 10
      pub idle_timeout_min: f64,           // default 30; 0 = disabled
      pub daily_budget_usd: Option<f64>,   // soft alert
      pub monthly_budget_usd: Option<f64>, // soft alert
      pub require_confirm_above_hourly: f64, // default 0.5
      pub require_typed_confirm_above_hourly: f64, // default 2.0
  }
  ```
- `crates/xrun-core/src/budget.rs`:
  - `enum DestroyReason { LifetimeCap, CostCap, IdleTimeout }`.
  - `evaluate_caps(inst: &Instance, now: DateTime<Utc>) -> Option<DestroyReason>`.
  - `accumulate_cost(inst: &Instance, now: DateTime<Utc>) -> f64` —
    `(now - created_at).hours * price_per_hour`.
  - `daily_spend(store, day: NaiveDate) -> Result<f64>` — сумма по
    runs.cost_usd_estimate где `created_at::date == day` + active по
    accumulated_cost.
- Тесты: матрица caps, граничные случаи (нет created_at, нет price), idle
  без `last_active_at` (фолбэк на created_at).

**Done when**: `cargo test -p xrun-core` зелёный, миграция применяется на
существующей БД без потерь данных.

### Stage 2 — Provision + daemon enforcement

- `xrun-vast/src/provision.rs`: при `create_instance` записывать caps в
  store (берём из manifest или из глобальных дефолтов). В manifest добавить
  опциональные поля `budget: { max_hours?, max_cost_usd?, idle_min? }`.
- `xrun-cli/src/commands/poll_daemon.rs`: на каждом тике для всех
  активных инстансов:
  1. Обновить `accumulated_cost` и `last_active_at` (по GPU util из
     `vast show instance`, если значение > порога).
  2. `evaluate_caps()` → если Some(reason): записать
     `auto_destroyed_reason`, эмитнуть event `instance.auto_destroyed`,
     вызвать `adapter.destroy_instance()`.
- Soft-alert: если `daily_spend > daily_budget_usd` — event
  `budget.daily_exceeded` (раз в день). Хард-стоп опционален через
  `daily_budget_hard: bool` (по умолчанию false).

**Done when**: интеграционный тест с фейковым адаптером проверяет
auto-destroy при `max_cost = 0.01` через 1 тик; при наличии recent
last_active_at idle-cap не срабатывает.

### Stage 3 — Confirm flow (CLI + TUI)

- `xrun-core::budget::ConfirmEstimate { gpu, hourly, max_hours, max_cost,
  balance, risk_tier }` — собирается из offer + caps + vendor_status.
- CLI: в `launch.rs` перед `provision` — `confirm_billable_or_exit`. Печать:
  ```
  Vast.ai · RTX 4090 · $0.42/hr
  Cap: 8h × $0.42 = $3.36 max
  Balance: $34.12   →   ETA at this rate: ~80h
  Confirm? [y/N]:
  ```
  При `risk_tier = TypedConfirm` — `Type 'launch' to confirm:`. `--yes`
  пропускает интерактив, но при non-tty без `--yes` → exit 1.
- TUI: `view/confirm_billable.rs` — modal с тем же контентом, цвет рамки
  по tier (green/yellow/red). Trigger из Launch picker и из command
  palette `:launch`.

**Done when**: попытка launch без tty и без `--yes` падает с понятной
ошибкой; в TUI модалка появляется и блокирует фоновые actions.

### Stage 4 — Spend dashboard + status_bar wiring

- Dashboard cards в `screens/runs.rs`:
  - `Burn`: суммарный `$/hr` по активным = sum(price_per_hour).
  - `Today`: `$ spent today` (готовые + accumulated по живым).
  - `Cap-left`: минимум `(max_cost - accumulated)` среди активных —
    показывает «сколько ещё может сгореть до auto-destroy».
- Status bar: при `daily_budget_usd` set и `daily_spend / budget > 0.8` —
  предупреждение `⚠ 84% daily`. При `balance < hourly_burn` —
  `⚠ <1h runway` красным.
- `count_up` анимация (уже есть из v0.2.2) для smooth-обновления чисел.

**Done when**: на запущенном инстансе dashboard в реальном времени
показывает растущий `Today` и убывающий `Cap-left`.

## Acceptance

1. `xrun launch exp/foo.yaml` без `--yes` в pipe-режиме — exits 1 с
   подсказкой про `--yes`.
2. Запустить инстанс с `--max-cost 0.05` на дешёвом GPU; через минуту
   poll-daemon его сносит, в `events.jsonl` запись `instance.auto_destroyed
   { reason: "cost_cap" }`, `xrun ls` показывает `auto-destroyed (cost cap)`.
3. TUI dashboard на живом инстансе показывает накапливающийся
   `Today: $X.XX` (обновляется ≥ раз в 10s) и убывающий `Cap-left`.
4. При `daily_budget_usd = 5.0` и достижении лимита — TUI status_bar
   подсвечивается, событие в `events.jsonl`. По умолчанию инстансы НЕ
   убиваются (soft); при `daily_budget_hard = true` — убиваются.
5. Confirm modal в TUI: для $2.5/hr GPU требует ввести `launch` для
   подтверждения; Esc отменяет без побочных эффектов.

## Out of scope

- **Forecasting** на основе истории похожих ранов — в v0.4 backlog.
- **Per-experiment budgets** (один манифест = один cap pool на N запусков) —
  отложено до v0.3.
- **Уведомления вне TUI** (email/desktop notify при auto-destroy) — backlog.
- **Запрет дешёвого оверрайда caps** — пользователь может пройти confirm,
  это намеренно; защита — не наручники.
