# xrun v0.2.2 — TUI UX polish: dashboard, animations, density

## Overview

После v0.2.1 интерфейс работает, но «ощущается дёшево»: половина экрана пустая,
вендорская информация спрятана за `V`, splash появляется только в first-run,
нет визуальной обратной связи на фоновые операции. Эта итерация — полировка
без новых фич.

## Pain points (явные и из кода)

### От пользователя
1. **Много пустого места**: 50/50-разбиение на Runs/Vendors/Instances оставляет огромные пустые блоки при 0-3 рядах.
2. **Splash при запуске не виден** — показывается только если `Credentials::is_empty()` (только в самый первый запуск).
3. **Вендорская инфа не на главном** — на Runs не видно ни балансa, ни статуса соединения.
4. **Нет анимаций**.
5. **Сухо, нечего смотреть**.

### Из аудита кода
6. Status bar — серая строка с одним сегментом, теряется внизу.
7. Throbber крутится только в Active runs; vendor-probe идёт молча.
8. Empty-states отсутствуют (`Active (0)` — просто пустая таблица).
9. Палитра plain-grey; `Theme::default()` и `high_contrast` обе невыразительны.
10. Сменa экрана — мгновенный jump cut без визуальной связки.
11. Vendors detail-pane занимает 50% даже когда в нём 3 строки.
12. На Instances/Remote табе данные появляются с задержкой ~500ms — выглядит как лаг.
13. `g g`/`G`/`,`/`V` нигде не подсказаны вне Help-оверлея.

## Context

- **Files involved (ожидаемая правка)**:
  - `crates/xrun-tui/src/theme.rs` — расширить палитру.
  - `crates/xrun-tui/src/view/status_bar.rs` — multi-segment, цветные иконки.
  - `crates/xrun-tui/src/view/runs.rs` — карточки сверху + адаптивные секции.
  - `crates/xrun-tui/src/view/instances.rs`, `vendors.rs` — адаптивные высоты, empty-states.
  - `crates/xrun-tui/src/view/splash.rs` — frame-based анимация, всегда показывать.
  - `crates/xrun-tui/src/app.rs` — анимационный таймер, всегда-splash, dashboard-cards state.
  - `crates/xrun-tui/src/state.rs` — поле `anim_frame: u64`, `last_balance_animation` snapshot для number-tween.
  - `crates/xrun-tui/src/view/anim.rs` (новый) — общие helper'ы (fade-in, count-up, char-reveal).
  - `crates/xrun-tui/src/view/cards.rs` (новый) — header-cards для Runs.
  - `docs/TUI.md` — мокапы новых экранов.

- **Constraints**:
  - Не тащить `tachyonfx` (TUI.md отдельным пунктом запрещает).
  - Не тратить CPU: анимации tick-driven (`render_interval = 100ms` уже есть), не free-running.
  - Не ломать существующие snapshot-тесты (`tests/snapshots/*.txt` → пересоздаются `BLESS=1`, но diff должен быть осмысленный).

## Development Approach

- Сначала **theme + status bar + empty-states** — самые видимые улучшения с малым кодом.
- Потом **adaptive layouts** — структурное.
- Потом **dashboard cards** + **always-on splash** — новые элементы.
- Анимации в конце: легче добавить когда layout стабилен.

Validation gate после каждой задачи:
- `cargo test --workspace` (snapshots могут потребовать BLESS).
- `cargo clippy --workspace --all-targets -- -D warnings`.
- Ручной запуск TUI: проверить, что не дёргается на 100ms-tick (CPU < 5%).

## Validation commands

- Tests: `cargo test --workspace` (snapshots: `BLESS=1 cargo test ...` если diff осмысленный).
- Lint: `cargo clippy --workspace --all-targets -- -D warnings`.
- Format: `cargo fmt --all -- --check`.

## Out of scope (для следующих v0.2.x / v0.3+)

- Полноценный сложный анимационный фреймворк (`tachyonfx` остаётся под запретом).
- Mouse support (хвоистая фича, отдельный плановый раунд).
- Layout customization пользователем (компактный/широкий режим).
- Charting в карточках (sparkline для balance over time) — нужен MLflow integration v0.3.
- Notifications/toast после завершения run-а — отдельная задача.
- Адаптация под маленький терминал (< 80 cols) — сейчас падает в кашу.

## Anti-patterns

- Анимации не должны крутиться вхолостую: фрейм инкрементится только когда `dirty || elapsed >= render_interval`. Не вводить `Animating` ветку в render loop, использующую `tokio::time::sleep(16ms)`.
- Не дублировать данные в карточках и в самих таблицах сверх меры — карточка показывает агрегат, таблица детали.
- Не делать splash длиннее 600ms для повторных запусков — раздражает.
- Не использовать `Color::Rgb()` без fallback'а: половина терминалов не поддерживает truecolor.
- Не рисовать transition если новый и старый экран одинаковые (`pop` на тот же).
- Не таскать `Instant` в `AppState` через `Clone` — заворачивать в `Arc`/`Cell` если нужно (или хранить как `u64` ms-since-start).
