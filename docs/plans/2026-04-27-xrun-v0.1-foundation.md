# xrun v0.1 — Foundation (workspace, manifest, sqlite, config, CLI scaffold)

## Overview

Заложить фундамент xrun без интеграции с vast.ai/Kaggle: cargo workspace из 3 крейтов
(`xrun-core`, `xrun-vast`, `xrun-cli`), парсер манифеста с детерминированным
canonical-hash, SQLite storage со схемой v1 и миграциями, конфиг в `~/.config/xrun/`,
типы событий и метрик, и clap-based CLI с заглушками-командами. После этого плана
бинарь `xrun` собирается, парсит манифесты, инициализирует БД/конфиг, и его команды
выводят полезный help. Реальный vast-адаптер и Python `xrun_hook` — отдельный
follow-up план.

## Context

- **Files involved**: всё новое — проект ещё пустой (только `README.md` и `docs/`).
- **Reference docs**:
  - `docs/ARCHITECTURE.md` — компоненты и crates
  - `docs/MANIFEST.md` — YAML-схема + правила хеширования
  - `docs/STATE.md` — SQLite-схема v1
  - `docs/EVENTS.md` — формат events.jsonl / metrics.jsonl
  - `docs/CLI.md` — список команд и флагов
- **Внешние зависимости**:
  - `clap` 4.x (derive), `serde`, `serde_yaml`, `serde_json`, `rusqlite` (bundled),
    `directories` (XDG/known-folders), `thiserror`, `anyhow`, `chrono`,
    `sha2`, `tracing`, `tracing-subscriber`, `ulid`.
  - Dev: `assert_cmd`, `predicates`, `tempfile`, `pretty_assertions`.
- **MSRV**: 1.78 (последний stable на момент старта).
- **Платформа**: dev и тесты — Windows + Linux. Пути через `Path::join` / `directories`.
- **Канонизация манифеста для hash** (фиксируем здесь, в docs нет): сериализация в
  JSON через `serde_json::to_value`, рекурсивная сортировка ключей объектов,
  удаление `null`-полей, числа в нормализованной форме (`Number::from_f64`),
  итоговый `to_string` без пробелов → `sha256` → hex lowercase. Определено в
  `xrun-core::manifest::canonical_hash` с unit-тестом на стабильность между запусками
  и платформами.
- **Конфиг каталог**: `directories::ProjectDirs::from("", "", "xrun").config_dir()`
  на Linux/macOS даёт `~/.config/xrun/`, на Windows — `%APPDATA%\xrun\config\`.
  Внутри: `config.toml` (несекретные) и `credentials.toml` (секреты, файл-permissions
  0600 на Unix).
- **Data каталог**: `ProjectDirs::data_dir()` — `~/.local/share/xrun/` или
  `%LOCALAPPDATA%\xrun\data\`. Содержит `runs.db` и `runs/<id>/`.

## Development Approach

- **Testing approach**: TDD для core-логики (canonical-hash, парсер манифеста,
  миграции SQLite). Regular для CLI-обвязки (clap parse — покрытие через
  `assert_cmd`).
- Каждая задача — один логический слой; завершается зелёным `cargo test --workspace`
  и `cargo clippy --workspace -- -D warnings`.
- **CRITICAL: every task MUST include new/updated tests.**
- **CRITICAL: all tests must pass before starting next task.**
- Внешние сервисы (vast.ai/Kaggle/MLflow) НЕ дёргаем в этом плане — все адаптеры
  получают только trait + stub-impl возвращающий `Err(NotImplemented)`.
- Никаких `unwrap()` в библиотечном коде; в тестах — допустимо.
- Никаких комментариев-WHAT; только WHY-комментарий, если что-то неочевидно.

## Validation commands

- Test: `cargo test --workspace`
- Lint: `cargo clippy --workspace --all-targets -- -D warnings`
- Format: `cargo fmt --all -- --check`

## Out of scope (для следующих планов)

- Реальные subprocess-вызовы `vastai` / `kaggle` (адаптеры в этом плане — стабы).
- Поллер (отдельный план — нужен после рабочего адаптера).
- Python `xrun_hook` пакет.
- TUI (`xrun-tui` крейт).
- MLflow REST клиент.
- `xrun sweep`, `xrun diff`, нотификации.

## Anti-patterns (явно НЕ делать)

- Не вводить `async` пока не нужен (v0.1 — sync; tokio появится с поллером в v0.1.bis).
- Не плодить trait'ы «на будущее»: `VendorAdapter` уже спроектирован — больше не добавлять.
- Не писать собственный yaml-парсер; serde_yaml достаточно.
- Не делать backwards-compat между migrations 001 и будущими — просто bump schema_version.
- Не реализовывать `--follow` в этом плане; явно exit 64 «not supported in v0.1».
